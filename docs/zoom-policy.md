# Zoom Policy

**実データ（北海道地方VBM Shapefile提供済み8火山、樽前山・有珠山 VLCM）に基づき確定。** 測定コマンドと結果は各節に記載。

## 現状の設定

- `scripts/build-vbm.sh`: `tippecanoe --force -P -n "Hokkaido VBM" ... -Z 5 -z 14`（`src/*_vbm.zip` に一致する全ファイルを結合して処理）。等高線・等深線系8コード（`7101/7102/7105/7106/7132/7133/7134/7135`）・道路4コード（`2101/2103/2106/2107`）・建物4コード（`3001/3002/3003/3004`）は feature 単位で `tippecanoe.minzoom=13` を付与
- `scripts/build-vlcm.sh`: `tippecanoe --force -P -n "Hokkaido VLCM" ... -Z 5 -z 14 -L natural:... -L artificial:...`（`src/*_vlcm.zip` に一致する全ファイルを結合して処理）

`-Z 5`（最小ズーム5）〜`-z 14`（最大ズーム14）で両方とも共通。`--drop-densest-as-needed` はデータ欠損のリスクがあるため不採用（2026-07-04 に試した上で撤回。詳細は後述）。密度対策は per-feature の `minzoom` シフトのみで行う方針。

## 実測結果

### VLCM: 警告なし

`work/vlcm/natural.ndjson` + `work/vlcm/artificial.ndjson`（計 1,594 features, 639,449 bytes）でローカル tippecanoe（v2.79.0）を実行 → タイルサイズ警告は **0件**。データ量が小さく、現状の `-Z 5 -z 14` のままで問題ない。

### VBM: ズーム 9〜12 で 5 タイルが 500KB 超過

`work/vbm/vbm_filtered.ndjson`（201,939 features, 71,124,591 bytes）でローカル tippecanoe を実行し再現:

```bash
tippecanoe --force -P -n "Tarumaezan VBM" -A "GSI" -N "tarumaezan-vbm" -Z 5 -z 14 -o /tmp/vbm_test.pmtiles work/vbm/vbm_filtered.ndjson
```

出力された警告（すべて `>500000` bytes、tippecanoe のデフォルト上限）:

| タイル (z/x/y) | サイズ (bytes) | 上限超過率 |
|---|---|---|
| 9/457/188 | 563,814 | +12.8% |
| 10/914/377 | 732,539 | +46.5% |
| 11/1829/755 | 673,527 | +34.7% |
| 11/1828/755 | 726,278 | +45.3% |
| 12/3658/1510 | 562,430 | +12.5% |

いずれも樽前山中心部と同じ座標域（`457/188` 系列はズームが上がるごとに `914/377`→`1828-1829/755`→`3656-3660/1508-1511` と同一地点にズームインした先）。原因は、VBM の全 63 種類の `分類コード` を単一の tippecanoe レイヤ（`-L` 未指定のためファイル名由来の1レイヤ）にまとめて詰め込んでいるため、中〜高密度ズームで等高線(3001 Polygon 112,444件 / 2101,2106 LineString 合計45,000件超）が同一タイルに集中すること。

### 2026-07-04 追記: 樽前山 + 有珠山の2火山結合での再測定

`just fetch-vbm usu` で有珠山データを取得し、樽前山と結合した状態（201,939 → 249,830 features、+47,891件）で `just build-vbm` を再実行した。

**結果: タイルサイズ警告は変化なし。** 上記と全く同じ5タイル・ほぼ同じサイズ（誤差数十bytes程度）のみが警告対象で、有珠山側から新たな警告は発生しなかった。理由は明快で、両火山は地理的に離れており、樽前山エリアの高密度タイルと有珠山エリアのタイルはズーム9以降では別々のタイル座標に分かれるため、単純結合では密度が加算されない。

これは「複数火山を結合すると密度過多エリアが増える」という当初の懸念に対する反証であり、少なくとも**重複しない火山同士の結合ではタイルサイズ警告は悪化しない**ことが実測で確認できた。悪化するとすれば、地理的に近接した火山同士（例: 樽前山と支笏湖対岸の風不死岳など、同一低ズームタイル内に収まる組み合わせ）の場合のみと考えられる。

**（2026-07-04 追記の追記）この「2火山では悪化しない」という結論は、8火山結合では覆った。** 下記「北海道地方8火山」の節を参照。少数サンプルでの結論を過度に一般化しないこと。

追加で、有珠山データ結合時に2件の新しい tippecanoe 警告が出たが、いずれも無害と判断:
- `ignoring dimensions beyond two`: 有珠山の等高線系ジオメトリに Z 座標（標高）が埋め込まれているため。tippecanoe は2Dタイル形式なので自動的に無視する（標高は別途 `標高` 属性としても保持されているため情報は失われない）
- `null geometry`: 有珠山データ中に3件、ジオメトリが null の feature（分類コード 5102×1, 5227×2）が存在した。全249,830件中0.001%程度で、GSI データ側の既知の欠損と考えられる。tippecanoe が警告を出しつつ自動的にスキップするため、パイプライン側の対処は不要

### 2026-07-04 追記: 北海道地方8火山（VBM Shapefile提供済み全火山）を結合した再測定

`just fetch-vbm` で北海道地方の VBM Shapefile 提供済み火山を全て取得（アトサヌプリ・雌阿寒岳・大雪山・十勝岳・樽前山・俱多楽・有珠山・北海道駒ヶ岳の8火山。恵山はVBM Shapefile未提供のため対象外）し、`just build-vbm` で結合した（249,830 → 549,748 features）。

**結果: 警告対象タイルが 5 → 14 件に増加した。** 新たにズーム6（1件）・ズーム8（1件）でも警告が発生し、ズーム11では7件のタイルが集中した（樽前山だけでなく、大雪山・十勝岳など内陸の山岳密集地帯が近接しているため、低ズームで複数火山が同一タイルに収まるようになった）。これは前回（2火山）の「地理的に離れていれば悪化しない」という結論を修正するもので、**北海道全域を結合すると密度過多エリアが増えることが実測で確認された**（意図的だったと分かった水深系コードの件と同様、少数サンプルでの結論は本格スケールで覆りうるという教訓）。

**対策: `--drop-densest-as-needed` を `scripts/build-vbm.sh`/`build-vlcm.sh` の tippecanoe 呼び出しに追加した。** 適用後、ビルドログには引き続き「tile ... size is ... >500000」という警告行が出力される（`--drop-densest-as-needed` が間引きを試行する過程で一時的に超過を記録するため）が、これは最終成果物の状態を表さない。**最終的な `dst/vbm.pmtiles` から `pmtiles tile <path> <z> <x> <y> | wc -c` で実際のバイト数を直接測定したところ、14候補タイル全てが500,000バイト未満に収まっていた**（最大は `11/1836/747` の497,051バイト）。

この経緯から、**ビルドログの `size is ... >500000` 行だけでは最終成果物の適合性を判断できない**ことが分かった。今後タイルサイズを検証する際は、ログの警告行数を見るのではなく、ビルド後に実際のタイルバイト数を測定すること（下記コマンド参照）。

### 2026-07-04 追記2: `--drop-densest-as-needed` を廃止し、等高線系コードの minzoom シフトに切り替えた

上記の「実測ゼロを確認」は**誤りだった**。`pmtiles tile <path> <z> <x> <y> | wc -c` は PMTiles に**格納されている gzip 圧縮後のバイト数**を返す。tippecanoe の `size is X >500000` 警告は**非圧縮のタイルサイズ**を指すため、比較する指標が違っていた（実際に該当タイルを `gunzip` して確認すると 1.2〜1.6MB あり、全く解決していなかった）。正しい検証には `pmtiles tile ... | gunzip -c | wc -c` で非圧縮サイズを見る必要がある（下記コマンド参照）。

加えて、目視確認により大雪山エリアで等高線（`7101`/`7102`）が `--drop-densest-as-needed` によって大量にドロップされている実害が見つかった。`--drop-densest-as-needed` は密度の高いタイルから特徴量を自動的に間引いてサイズ制限内に収めようとするが、**どの feature が失われるかは制御できず、望まないデータ欠損を生む**。tippecanoe-decode で問題タイルの内訳を調べたところ、`7102`(主曲線)・`7101`(計曲線) がバイト数の大半（あるタイルでは7102だけで非圧縮換算3.6MB相当）を占めていることを確認した。

**対策として `--drop-densest-as-needed` を完全に廃止し、代わりに等高線・等深線系8コード全て（`7101/7102/7105/7106/7132/7133/7134/7135`）の `minzoom` を 11→12 に統一した**（データの間引きは一切発生しない。単に低いズームでは描画されなくなるだけで、feature数は549,748件のまま不変）。

結果:
- 大雪山エリアの警告タイル（旧: z11で7件）は**全て解消**
- 樽前山エリアに別原因の4タイル（z8: `228/94`, z10: `914/377`, z11: `1829/755`, z12: `3658/1510`）が残った。`tippecanoe-decode` で内訳を確認したところ、**等高線とは無関係**で、`2101`/`2106`/`5101`/`5102`（LineString、具体的に何を表すか未特定。属性は `注記`/`表示区分`程度のみ）が北海道全域にわたって高密度に存在していることが原因と判明。これらのコードには現状 minzoom 制限が無い（`-Z 5` の全体設定のまま）。詳細・対応方針は `docs/schema.md` の該当セクション参照——**未解決のまま次セッションに持ち越し**

### 2026-07-04 追記3: 分類コードの正体特定 → 道路・建物にも同じ minzoom シフトを適用し、全警告を解消

`2101`/`2106`/`5101`/`5102` の正体を、樽前山のシェープファイルを1つずつ変換して確認したところ判明した（詳細は `docs/schema.md`）:
- `2101`/`2103`/`2106`/`2107` = **道路**（8火山結合で2101=88,615件・2106=20,114件）
- `5101`/`5102`/`5106` = **水涯線・海岸線**

ユーザーより、[国土地理院最適化ベクトルタイル](https://github.com/gsi-cyberjapan/optimal_bvmap)（`style/std.json`）の設計を参考にする方針を得た。実際に確認すると、GSIの設計では主要道路（国道・高速）はズーム4-10から、詳細な道路表現はズーム11から表示され、市区町村界もズーム11から、という段階設計になっている。一方、建物は面積による段階的な間引きが行われているが、**今回はそこまでの作り込みは行わない方針**とした（ユーザー指示）。海岸線・水域はGSIの設計でも明示的なズーム制限が無く、低ズームから表示される。

方針: 道路（`2101/2103/2106/2107`）・建物（`3001/3002/3003/3004`）にも、等高線と同様の単純な `minzoom` シフトを適用する。複雑な面積ベースの段階的間引きは行わない。海岸線（`5101/5102/5106`）はGSIの設計に倣い、意図的に minzoom 制限を加えなかった。

**適用と実測（`pmtiles tile | gunzip -c | wc -c` で非圧縮サイズを都度確認しながら反復）:**

1. 道路・建物を `minzoom=12`（等高線と同じ値）に設定 → 警告タイルが14件→1件（`12/3658/1510`、樽前山火口周辺）まで減少。ただし tippecanoe のビルドログは警告0件を報告していたにもかかわらず、実際に非圧縮サイズを測定すると556,727 bytesで**依然として超過**していた。ログの「警告0件」だけを信用してはいけないことが改めて確認された（`--drop-densest-as-needed` を使っていなくても、ログとの乖離が起こりうる）
2. この最後の1タイルは、等高線(7101/7102)と道路(2101/2106)がズーム12で同時に出現し重なったことが原因。ユーザーの「1段階で不十分なら2段階シフトしてよい」という方針に従い、**等高線・道路・建物の全てを `minzoom=13` に統一** → 全13候補タイルを `gunzip` 後の非圧縮サイズで直接再測定し、**全て500,000バイト未満に収まることを確認**（最大は `8/228/94` の472,270 bytes）

最終的な `minzoom` 設定（`scripts/build-vbm.sh`）: 等高線・等深線系8コード + 道路4コード + 建物4コードの計16コードが `minzoom=13` で統一。データの間引き・欠損は一切無し（549,748 features のまま不変）。海岸線・行政界・その他のコードは無制限（`-Z 5` のまま）。

### 現時点の判断

- `--drop-densest-as-needed` は不採用（データ欠損のリスクがあるため）。密度対策は全て per-feature の `minzoom=13` シフトのみで行う
- 北海道地方VBM Shapefile提供済み8火山の結合データで、タイルサイズ警告は実測ゼロを確認済み（ログではなく `gunzip` 後の非圧縮サイズで検証したもの）
- 恵山のVLCM追加、または本州側火山を追加結合する際は、同じ手順（下記の非圧縮サイズ実測）で再検証すること。特に**ビルドログの警告件数だけで判断せず、必ず `gunzip` 後の実測を行うこと**（ログが「警告0件」でも超過していたケースが実際にあった）

### タイルサイズの正しい検証手順（圧縮後のサイズではなく非圧縮サイズで判断）

```bash
just build-vbm

# ビルドログに出る "size is X >500000" は非圧縮サイズを指す。
# pmtiles tile はPMTilesに格納された圧縮後のバイト数を返すため、
# gunzip して非圧縮サイズに戻してから比較すること:
pmtiles tile dst/vbm.pmtiles <z> <x> <y> | gunzip -c | wc -c
```

## VBM/VLCM 共通の再測定手順

新しい火山データを追加した後、または本番相当のデータ量になった時点で、以下を再実行してタイルサイズ警告を確認すること。

```bash
just build-vbm 2>&1 | grep -E "size is.*>500000"
just build-vlcm 2>&1 | grep -E "size is.*>500000"
```

警告が増加・悪化した場合は、上記「現時点の判断」の対策リストから着手する。
