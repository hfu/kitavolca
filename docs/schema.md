
### 2026-07-04 追記: 分類コード → 由来シェープファイル名（実際の意味）の対応

これまで `分類コード` の実際の意味（何を表す図式か）を特定できていなかったが、樽前山の ZIP 内シェープファイルを1つずつ変換して `分類コード` を確認し、直接対応付けた。

測定コマンド:
```bash
gdal vsi list -R -f json "/vsizip/$(pwd)/src/tarumae_vbm.zip" | jq -r '.[] | select(test("\\.shp$"; "i"))' | while read -r shp; do
  name=$(basename "$shp" .shp)
  codes=$(ogr2ogr --config SHAPE_ENCODING CP932 -f GeoJSONSeq /vsistdout/ "/vsizip/$(pwd)/src/tarumae_vbm.zip/$shp" 2>/dev/null \
    | jq -r '.properties["分類コード"]' | sort -u | tr '\n' ',')
  echo "${name}: ${codes}"
done
```

### 保持する属性（確定）

`scripts/build-vbm.sh` の実装通り、以下を採用する。

- **削除**: `ID番号`（feature 内シーケンス番号。スタイリング・クエリに無価値なため削除）
- **保持（そのまま）**: `出典コード`, `出典レベル`, `標高`, `水深`, `水深値`, `名称`, `注記`, `表示区分`, `三角点標高`, `水準点標高`, `分類コード`
  - いずれも小さいスカラー値で、タイルサイズへの影響は軽微
  - `分類コード` は `properties` 内にも残したまま、`tippecanoe.layer`（文字列化）としても複製する。フィルタ用に文字列比較しやすくするため
- **追加**: feature 直下の `tippecanoe.layer`（`分類コード` の文字列版）。`tippecanoe.minzoom = 11` を等高線・等深線系コードにのみ付与

### 2026-07-04 追記: minzoom を全等高線・等深線系コードで 12 に統一（`--drop-densest-as-needed` 廃止）

北海道8火山を結合した際、大雪山エリアで等高線（`7101`/`7102`）が大量にドロップされる問題が目視で見つかった（`--drop-densest-as-needed` によるもの）。原因を `tippecanoe-decode` で分析したところ、該当タイルの内容は `7102`(主曲線)・`7101`(計曲線) がバイト数の大半を占めていることを確認。

対策として `--drop-densest-as-needed` を完全に廃止し、代わりに等高線・等深線系8コード全て（`7101/7102/7105/7106/7132/7133/7134/7135`）の `tippecanoe.minzoom` を統一した（旧: `7102/7106/7133/7135` のみ `minzoom=11`、`7101/7105/7132/7134` は無制限という不整合な設計だった）。これによりデータの間引き・欠損は一切発生しない（feature数は549,748件のまま不変）。

結果: 大雪山エリアの警告タイル（z11で7件）は全て解消。樽前山エリアには別原因（後述）で4タイルの警告が残るが、等高線とは無関係と判明した——この残りの問題は道路・建物コードの正体特定（後述「2101/2106/5101/5102の高密度」節）と合わせて解決し、**最終的に minzoom は等高線・道路・建物の計16コードで 13 に統一**された（経緯は `docs/zoom-policy.md` 参照）。

これにより、下記「未解決事項（水深系）」の対象コード分け自体は設計判断として維持しつつ、**minzoom は8コード共通で13** という運用に変更された。以下の分析（計曲線/主曲線のペア構造の実測根拠）自体は変更なし、参考情報として残す。

### 等高線・等深線コードの minzoom=11 対象選定の根拠（実測により確定。2026-07-04 時点で minzoom の値自体は13に変更されたが、コード間のペア構造分析は有効）

`7101/7102`, `7105/7106` は同じ属性形状（標高を持つ LineString）だが、`分類コード` ごとの標高値の分布を実測すると、明確な「計曲線（強調線・間引かれた線）/ 主曲線（通常線・密な線）」のペア構造になっていることがわかった。

測定コマンド:
```bash
jq -r 'select(.properties["分類コード"]==<code>) | .properties["標高"]' work/vbm/all_raw.ndjson \
  | awk '{tot++; if ($1 % 25 == 0) m25++} END{print m25"/"tot}'
```

結果:

| 分類コード | 件数 | 標高が25の倍数の割合 | 解釈 |
|---|---|---|---|
| 7101 | 3,723 | 3722/3723 (99.97%) | **計曲線**（25m間隔の強調線） |
| 7102 | 16,837 | 1/16,837 (0.01%) | **主曲線**（5m間隔。7101と重複する25の倍数を意図的に除外） |
| 7105 | 90 | 80/90 (88.9%) | 補助曲線系の計曲線 |
| 7106 | 824 | 3/824 (0.36%) | 補助曲線系の主曲線 |

7102 は7101（25の倍数）をほぼ完全に除外しており（16,837件中1件のみ例外）、7101と7102が「同じ地形を、間引いた強調表示用（計曲線）」と「全間隔表示用（主曲線）」に意図的に分割していることが確認できる。7105/7106も同型のペア。

**結論**: `minzoom=11` が付与されている `7102, 7106` は各ペアの「主曲線（密度が高く、低ズームでは線が密集しすぎる）」側であり、`7101, 7105`（計曲線・間引き済みで密度が低い）は同じ理由でのズーム制限が不要なため対象外——という設計は密度データと整合しており、見落としではなく合理的な選択だったと判断できる。

### 未解決事項（水深系: 7132/7133/7134）

等深線側にも同様の計曲線/主曲線ペア構造がある可能性が高いが（`7133` のみ `minzoom=11` 対象）、樽前山データでは該当件数が極端に少なく（7132: 9件, 7133: 13件, 7134: 3件）、標高系と同じ手法で統計的に裏付けるには不十分。水深値も 0.25/0.5/0.63/0.88 とバラつきがあり、単純な倍数関係では説明しきれない。

- 海岸線・湖沼を含む火山（例: 支笏湖に近いエリア、有珠山の噴火湾側など）のデータで再測定し、同じ「間引き倍数関係」が成立するか確認すること
- 2026-07-04 時点で minzoom は8コード共通で13（上記追記を参照）。コード分けの倍数関係の検証自体は引き続き未解決

**2026-07-04 追記**: `just fetch-vbm usu`（有珠山、噴火湾に面する）で実際に検証を試みたが、有珠山の VBM Shapefile ZIP（`vbm07-2022-shp.zip`）には `等高線.shp` / `等深線.shp` に相当するファイル自体が含まれておらず（17ファイル中に無し）、標高・水深系コード（7101-7106, 7132-7135）を1件も含まないことが判明した。海岸に近い火山だからといって等深線データが含まれているとは限らない（マップシートの版・範囲による）。水深系コードの検証には、実際に等深線を含むことを事前に確認した火山データが必要——単に「海に近い」だけでは不十分。この件は未解決のまま持ち越す。

### 分類コード 2101/2106/5101/5102 の正体特定と対応（2026-07-04 解決）

大雪山の等高線問題を修正した後、樽前山エリアに残った4件のタイル警告（z8/10/11/12）を `tippecanoe-decode` で調べたところ、等高線とは無関係で、`2101`・`2106`・`5101`・`5102` が北海道全域にわたって非常に高密度に存在していることが原因と判明した。例えばズーム8のタイルでは `2101` だけで36,677件・6.8MB相当（decoded JSON換算）に達する。

正体は、樽前山のシェープファイルを1つずつ変換して `分類コード` を直接確認して判明した（詳細な対応表は上記「分類コード → 由来シェープファイル名」参照）:
- `2101/2103/2106/2107` = **道路**
- `5101/5102/5106` = **水涯線・海岸線**

ユーザーの指示で[国土地理院最適化ベクトルタイル](https://github.com/gsi-cyberjapan/optimal_bvmap)の設計（`style/std.json`）を参考にした。GSIでは主要道路がズーム4-10、詳細な道路はズーム11から、建物は面積による段階的な間引きを行っているが、**今回は建物の面積間引きのような作り込みは行わない方針**とし、道路・建物にも等高線と同じ単純な `minzoom` シフトを適用した。海岸線はGSIの設計でも明示的な制限が無いため、意図的に `minzoom` を付与しなかった。

最終結果（`docs/zoom-policy.md` に詳細）: 等高線・等深線系8コード + 道路4コード（`2101/2103/2106/2107`）+ 建物4コード（`3001/3002/3003/3004`）の計16コードを `minzoom=13` に統一。北海道8火山結合データで、タイルサイズ超過（非圧縮バイト数で実測）はゼロになることを確認済み。

## VLCM Schema

### 元属性

`work/vlcm/natural.ndjson`, `work/vlcm/artificial.ndjson` を実測（`jq -r '.properties | keys | join(",")' | sort -u`）。

- `ID`, `code1`, `code2`, `code3`, `code4`, `name` のみ。全レイヤ共通で、VBM のような分岐はない
- `code1`〜`code4` は階層的な分類コード（桁数が増えるほど詳細）。例: `code1=1, code2=101, code3=10101, code4=1010101, name=火口跡`
- `name` はすでに日本語の人間可読ラベル（VBM の数値 `分類コード` と異なり、別途コード表を参照しなくても意味がわかる）

### 保持する属性（確定）

- **全属性をそのまま保持**（フィルタ不要）。属性数が少なく（6キー固定）、`name` が既に人間可読なため、削除・変換の必要性がない

### natural / artificial の実測件数（樽前山データ）

| source-layer | 件数 | ジオメトリ内訳 |
|---|---|---|
| natural | 1,581 | Polygon 1,556 / LineString 25 |
| artificial | 13 | Polygon 13（全件 `code2=105`, `name=人工改変地`） |

`artificial` が極端に少ないのは樽前山エリアの人工改変地が実際に少ないためで、パイプラインの不具合ではない（ファイル名 `shizen`/`jinko` によるレイヤ振り分けは正しく機能している）。

### 2026-07-05 追記: VLCM の公式配色を調査し `docs/style.json` に反映

火山土地条件図の色は、国土地理院防災ポータルの凡例ページ（https://disaportal.gsi.go.jp/maps/image/legend/vlcd.html）から火山ごとの凡例画像（例: 樽前山 `https://maps.gsi.go.jp/legend/l_vlcd_10tarm.jpg`）をダウンロードし、Python（Pillow）でスウォッチ部分のピクセルを平均してRGB値を実測した。

**重要な発見**: 色は `code2`/`code3`（分類コード階層）単位ではなく、**`name`（地形名）単位で個別に割り当てられている**。例えば `code3=10101`（火口グループ）に属する「カルデラ壁」「火口原」「火口壁」は、同じ code3 でありながら全く異なる色（水色・クリーム・ピンク）が使われている。そのため、`docs/style.json` では `name` 属性に対する `match` 式で配色し、名称ごとの色を1つずつ実測値で埋めた（火口跡・火口・階段状地形など、凡例上ライン記号のみで塗り色を持たないものは `fill-opacity: 0` + 赤系のラインとして再現）。

「1739年火砕流堆積地形」のような噴火年を含む地名は樽前山・支笏火山に固有で、他火山では別の名称になる。有珠山のデータ（`just fetch-vlcm usu`）で実際に確認したところ、樽前山の凡例に無い名称（「馬蹄形カルデラ形成以前の火山体斜面」「潜在円頂丘」「スコリア丘」等）が多数存在した。これらは樽前山凡例の類似カテゴリ（成層火山体斜面・溶岩円頂丘・火砕丘等）から類推した色を暫定的に割り当てている（`docs/style.json` 生成に使ったスクリプトの `NAME_COLORS` コメント参照）。未知の `name` に対しては `code2` ベースのフォールバック色（101=火山地形系, 102=基盤山地, 103=堆積地形, 104=断層, 106=水部・ガリ）を用意し、さらに未知の場合は灰色にフォールバックする。

今後別の火山を追加した際、樽前山の凡例に無い `name` が出てきたら、その火山自身の公式凡例画像（`disaportal.gsi.go.jp` のページから辿れる）を確認し、実測値で `docs/style.json` の `NAME_COLORS` 相当のテーブルを更新すること。

**2026-07-05 追記（ポリゴン境界線の低ズーム対策）**: 実データで確認したところ、`名称="火口"` は（他の火口跡・階段状地形等と違い）実際には Polygon ジオメトリだった。低ズームでは1,500件超のPolygon境界線が重なって黒く潰れて見える問題があったため、`vlcm-natural-line` を3レイヤーに分割した:
- `vlcm-natural-crater-outline`: `火口`（Polygon）専用、赤線、ズーム制限なし
- `vlcm-natural-boundary`: その他の Polygon 境界線、`minzoom: 11`（低ズームでは非表示にして黒つぶれを回避）
- `vlcm-natural-symbol-line`: 実際に LineString 型の地形記号（火口跡・階段状地形・断層等）、赤/黒線、ズーム制限なし

## Refinement Process（今後の火山データ追加時）

1. 新しい火山の ZIP を `src/` に配置し `just inspect` で概要確認
2. `just build-vlcm` / `just build-vbm` を実行
3. 上記と同じ `jq` コマンドで属性キー・分類コード分布を実測し、本ドキュメントの表と差分がないか確認
4. 差分があれば（未知の分類コードなど）、本ドキュメントと `scripts/build-vbm.sh` を更新
